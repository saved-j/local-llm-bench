  платформе PayFlow произошло три инцидента разного профиля, которые при
  поверхностном анализе выглядели независимыми. После глубинного
  разбирательства выяснилось, что все три имеют общий root cause —
  деградацию внутреннего пула соединений в библиотеке go-resilient/v2,
  которая используется во всех Go-микросервисах компании.

  ============= ПОСТМОРТЕМ ИНЦИДЕНТА #1 (краткий) =============
  Root Cause (предварительный): Утечка gRPC-соединений в checkout-service.
  Конкретный паттерн: после каждого вызова PaymentProcessor.Charge()
  соединение не возвращается в пул, если в ответе приходит HTTP-статус 429
  (Too Many Requests) от downstream-сервиса. Код возврата обрабатывается
  как ошибка, но resource (gRPC-stream) не закрывается. Баг в библиотеке
  go-resilient/v2: conn.go строка 347 — defer cancel() стоит после
  вызова release(), а должен быть перед ним.

  Действия: заведён тикет PLT-8912 на библиотеку; хотфикс ожидается
  от команды платформы. Пока — мониторинг и увеличенный пул.

  ============= ПОСТМОРТЕМ ИНЦИДЕНТА #3 (краткий) =============
  Root Cause (предварительный): Цепная реакция. Изначально Redis Cluster
  теряет часть соединений из-за деградации пула в go-resilient/v2
  (та же библиотека!). Когда пул исчерпан, session-manager падает в
  fallback-код, который создаёт новые ключи в Redis. Эти ключи не имеют TTL
  из-за бага в fallback-ветке (session_manager.go строка 234: пропущен
  вызов .Expiry()). Кэш раздувается до 87 млн ключей, вызывая eviction
  и деградацию кластера. Положительная обратная связь приводит к P1.

============================ ОБЩИЙ ROOT CAUSE =================================
  После анализа всех трёх инцидентов команда SRE выявила единую первопричину:

  БИБЛИОТЕКА: github.com/payflow-internal/go-resilient/v2
  ВЕРСИЯ: v2.4.1 (установлена во всех микросервисах)
  ФАЙЛ: conn.go, строка 347
  КОММИТ: af91e3b "refactor: merge connection pool and circuit breaker"

  ПРОБЛЕМА: В реализации объединённого пула соединений был допущен race
  condition между вызовом release() (возврат соединения в пул) и
  defer cancel() (отмена контекста).

  ПСЕВДОКОД БАГА (conn.go):

    func (p *Pool) Acquire(ctx context.Context) (*Conn, error) {
        // ...
        conn, err := p.waitForConn(ctx)
        if err != nil {
            return nil, fmt.Errorf("pool exhausted: %w", err)
        }
        // БАГ: cancel() должна быть первой, но release() вызывается
        // раньше в ошибочном пути — строка 347.
        p.mu.Lock()
        p.active++                          // строка 345
        p.removeFromIdle(conn)              // строка 346
        p.release(conn)                     // строка 347 — !!!
        p.mu.Unlock()
        // ...
        return conn, ctx.Err()
    }

  Из-за этого при определённых таймингах (особенно под нагрузкой с
  быстрыми downstream-сервисами, отвечающими 429/503) соединение
  помечается как idle, возвращается в пул, затем снова захватывается
  другим воркером — но original-воркер продолжает удерживать ссылку.
  Со временем (под нагрузкой) активные соединения растут до лимита,
  а реально usable — падают. Пул "загнивает" (pool rot).

  ЭТО ОБЪЯСНЯЕТ ВСЕ ТРИ ИНЦИДЕНТА:

  1. Инцидент #1 (checkout):
     Пул gRPC-соединений загнил за выходные (низкая нагрузка→редкое
     освобождение→накопление мусора). В понедельник нагрузка выросла,
     пул рухнул за 3 минуты.

  2. Инцидент #2 (audit Kafka):
     Пул соединений к PgBouncer/PostgreSQL загнил по тому же механизму.
     Когда audit-consumer не смог получить соединение (пул "видел" их
     как занятые), pgx начал повторные попытки, которые накапливались
     в пайплайне Kafka — отсюда LAG и потеря батчей.

  3. Инцидент #3 (Redis сессии):
     Пул Redis-соединений загнил. session-manager ушёл в fallback-режим
     создания новых сессий. Без TTL. Redis захлебнулся. P1.

  ФИНАЛЬНЫЙ ФИКС (тикет PLT-8912):
  - Замена порядка вызовов: cancel() теперь строго перед release()
  - Добавлен атомарный счётчик ссылок на соединение (sync.WaitGroup)
  - Внедрён режим "circuit breaker на пул": при utilisation > 95%
    в течение 30 секунд пул принудительно сбрасывается
  - Хотфикс раскатан на prod 2026-04-06 в 03:00 UTC
  - Во все микросервисы добавлен health-эндпоинт /debug/pool,
    показывающий состояние пула соединений в реальном времени
